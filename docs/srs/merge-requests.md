# SRS: Merge Requests Dashboard

## 1. Introduction
The Merge Requests screen provides a unified view of all pending code integrations across GitLab, allowing users to track the lifecycle of agent-contributed code from proposal to merge.

## 2. Access Control
- **Roles**: Developers, QA, PO, Admins.
- **Permissions**:
  - `PO` / `Admin`: Can trigger manual sync and review all MRs.
  - `Dev`: Can view and sync MRs for their projects.

## 3. UI Components
- **Dashboard Stats**: Summary of Open, Merged, and Closed MRs.
- **MR Data Table**: List with Title, Project, Author (AI/Human), and GitLab Status.
- **Sync Header**: Global "Sync with GitLab" button.

## 4. Functional Requirements

### [SRS-MRD-001] Fetch Integrated MR List
- **Trigger**: Screen navigation.
- **Input**: Project filter (optional).
- **Output**: List of locally cached MR metadata.
- **System Logic**: Hiện backend expose task-scope endpoint `GET /api/v1/tasks/:id/gitlab/merge_requests`; màn hình tổng hợp cần gom theo task/project ở tầng client.
- **Validation**: Show GitLab connection status if the API returns an integration error.

### [SRS-MRD-002] Manual GitLab Sync
- **Trigger**: Click "Sync with GitLab".
- **Input**: None.
- **Output**: Progress indicator -> Refreshed list.
- **System Logic**: Hiện sync chính đi qua webhook ingestion `POST /api/v1/webhooks/gitlab` từ phía GitLab; UI manual-sync endpoint riêng chưa có.
- **Validation**: Handle GitLab API rate limiting with a 429-aware retry or alert.

### [SRS-MRD-003] Deep Link to GitLab
- **Trigger**: Click "View on GitLab" link in row.
- **Input**: `gitlab_url`.
- **Output**: External browser tab opened.
- **System Logic**: Standard link behavior.

### [SRS-MRD-004] Filter and Search
- **Trigger**: UI interaction with filters.
- **Input**: `status`, `project`, `author_type`.
- **Output**: Refined table view.
- **System Logic**: Backend filtering via query parameters or client-side if dataset is small.

### [SRS-MRD-005] Webhook Status Verification
- **Trigger**: Background polling or screen load.
- **Input**: None.
- **Output**: Indicator showing if the project's GitLab webhook is healthy.
- **System Logic**: Dùng `GET /api/v1/projects/:id/gitlab/status` để kiểm tra trạng thái cấu hình GitLab hiện có.

## 5. Non-Functional Requirements
- **Reliability**: Any sync failure must be captured in the platform logs for debugging connectivity issues between ACPMS and GitLab.
- **User Experience**: Provide visual indicators for MRs generated by the AI agent vs. those from human contributors.
